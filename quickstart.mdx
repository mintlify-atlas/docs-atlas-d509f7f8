title: "Get started with Shaiya Core"
description: "Set up Shaiya Core server and client modules from a clean checkout in four steps, including prerequisites, build commands, and runtime file setup."
---

This guide walks you through building both the server module (`sdev`) and the client module (`sdev-client`) from a clean checkout, then placing the runtime configuration files that `sdev` needs to start correctly. By the end you will have compiled DLLs ready to load alongside the official Shaiya server and client binaries.

## Prerequisites

Before building, make sure the following are installed and available on your `PATH`:

- **Windows 10** or newer
- **Visual Studio 2022** with the Desktop development with C++ workload
- **C++23** support (included with Visual Studio 2022 17.x)
- **x86 build target** — all modules compile for 32-bit
- **Microsoft Visual C++ Redistributable (x86):** [https://aka.ms/vs/17/release/vc_redist.x86.exe](https://aka.ms/vs/17/release/vc_redist.x86.exe)
- **Microsoft DirectX SDK (June 2010)** — required for the client module only

## Steps

<Steps>
  <Step title="Clone or download the repository">
    Get the source from the Shaiya-Core GitHub repository. Use a clean checkout for any build intended as a public release, because build outputs are excluded from version control and should not be mixed with working-directory changes.

    ```bash
    git clone https://github.com/Spelunkern/Shaiya-Core.git
    cd Shaiya-Core
    ```
  </Step>

  <Step title="Build the server module">
    Run MSBuild targeting the `sdev` project in Release|x86 configuration. This produces `sdev.dll`, which is loaded into `ps_game.exe`.

    ```powershell
    MSBuild.exe .\Shaiya-Core.sln /t:sdev /p:Configuration=Release /p:Platform=x86 /m
    ```

    The `/m` flag enables parallel builds. The output is written to `Release\sdev.dll` in the repository root.
  </Step>

  <Step title="Build the client module">
    Use the provided wrapper script to build `sdev-client`. The script resolves MSBuild automatically through `vswhere` or known install paths, restores the DirectX SDK package dependency, and builds `sdev-client` as Release|x86.

    ```powershell
    .\build-client.cmd
    ```

    You can also use the PowerShell equivalent:

    ```powershell
    .\build-client.ps1
    ```

    Both scripts default to `Release` configuration and the `Build` target. The output is `Release\sdev-client.dll`.
  </Step>

  <Step title="Place runtime files">
    `sdev` resolves paths from the game service executable directory and then the `Data/` subfolder. Create a `Data/` folder alongside `ps_game.exe` and place the following files inside it before starting the server:

    | File | Purpose |
    |---|---|
    | `Data/ServerConfig.ini` | Global limits: `LevelCap` and `EnchantCap`. **Created with defaults when missing.** |
    | `Data/BattleFieldMoveInfo.ini` | Battleground level ranges and per-family destination coordinates. |
    | `Data/SetItem.SData` | Decrypted server-side item set synergy data. |
    | `Data/ChaoticSquare.ini` | Item synthesis and chaotic square recipes. |
    | `Data/RewardItem.ini` | Timed account reward item event list. |
    | `Data/Roulette.ini` | Roulette token ID and reward table. |

    `ServerConfig.ini` is the only file that is auto-created with defaults on first launch. All other files must be provided before the features that depend on them are active.
  </Step>
</Steps>

<Tip>
Build outputs (`Release/`) are excluded by `.gitignore`. Always build from a clean checkout when preparing files for a public release, and test the outputs against the matching `ps_game.exe` and `Game.exe` binaries before distributing.
</Tip>

<Note>
Each module has its own README with feature details, configuration examples, and compatibility notes. Read `sdev/README.md` before making server-side changes and `sdev-client/README.md` before modifying client behavior.
</Note>
