Shaiya Core implements a set of custom item effects in `sdev` that extend the standard item effect system. These effects are handled server-side by `hook::item_effect()` and provide features not present in the stock server. The full `ItemEffect` enum is defined in `shaiya/include/common/ItemEffect.h`; the entries below focus on the effects that `sdev` actively intercepts and processes.

## Recreation rune notes

HP, MP, and SP recreation (effects 220–231) is restricted to non-weapon, non-accessory armor-style items. Weapons and accessories are rejected before the recreation logic runs.

The item's `ReqWis` field is reused as the maximum allowed value for a single craft stat. The server reads this value to clamp random and perfect recreation results.

MP recreation uses the `maxMana` item field; SP recreation uses the `maxStamina` item field. This distinction is important when reading or writing item data structures.

The client accepts effects 220–240 in the blacksmith recreation window, allowing players to place these items in the appropriate slot. The server performs the actual craft-stat mutation; the client only gates the window placement.

## Safety charms (effect 103)

Safety charm behavior is integrated into both the lapisian enchant success and failure code paths. When a safety charm item is present in the relevant inventory slot, item destruction on enchant failure is suppressed.

The effect corresponds to the `LapisianAddItemProtect` enum value and uses the same slot validation as standard lapisian operations. Packet `0x805` (lapisian add) carries an `itemProtect` flag in EP6 and EP6.4 variants, which the server checks in conjunction with the charm item.

## Ability transfer cube (effect 105)

Effect 105 (`ItemAbilityTransfer`) transfers craft identity from one item to another. The inventory bag layout the server expects is:

- **Slot 1** — source item (the item whose craft data will be moved away)
- **Slot 2** — the transfer cube itself (consumed on use)
- **Slot 3** — target item (the item that receives the craft data)

The `CraftName` and `Gems` fields are moved from the source item to the target item. The source item's craft fields are cleared after the transfer. The result is sent to the client via packet `0x811` (`GameItemAbilityTransferOutgoing`).

## Chaotic square and synthesis

Chaotic square and item synthesis effects are handled through configured recipe tables rather than hardcoded effect IDs. Recipes are loaded at server startup via `Configuration::LoadItemSynthesis()` from `Data/ChaoticSquare.ini`.

The client opens the chaotic square window with packet `0x830`, requests a specific synthesis recipe with `0x831`, submits a synthesis attempt with `0x832`, and submits a free-form synthesis with `0x833`. The server responds with `GameItemSynthesisOutgoing` (`0x832`) or `GameItemFreeSynthesisOutgoing` (`0x833`) carrying a `Success` or `Failure` result.

For configuration format and recipe examples, see [Item synthesis configuration](/configuration/item-synthesis).
